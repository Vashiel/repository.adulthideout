# AdultHideout 1.0.14 Todo - Bugfix and Optimization

Codename: Featherlight

Goal: 1.0.14 should be a calm bugfix and optimization release after the larger 1.0.13 expansion. The cleanup target is the whole addon, not only `resources/lib`.

## Priority A - Official Source Check

- Soften the "unofficial installation?" warning when only the online metadata check fails because of DNS, timeout or temporary GitHub/network errors.
- Keep local file hash verification strict: if bundled `resources/official_hashes.json` matches the installed files, the addon should not scare the user with an unofficial-installation dialog just because GitHub could not be reached.
- Suggested behavior:
  - Repository/addon metadata mismatch: warning remains.
  - Local hash mismatch: warning remains.
  - Remote metadata unavailable but local hashes pass: log warning only, or show a small non-blocking notification.
  - Remote metadata unavailable and local hashes fail: show full warning.
- Add better log wording so the user can distinguish "network check unavailable" from "installation modified".
- Test cases:
  - Normal online check succeeds.
  - GitHub/DNS unavailable, local hashes pass.
  - GitHub/DNS unavailable, local hash mismatch.
  - Official repository addon missing or wrong.

## Priority A - Whole Addon Cleanup and Size Audit

- Audit the complete addon tree for old leftovers, unused files, duplicate helpers, stale generated files, dead logos, obsolete settings and abandoned site/resolver code.
- Keep the zero-dependency direction: prefer bundled pure-Python helpers and local resolver code over new external package requirements.
- Start in `resources/lib` because it has the highest shared-risk and size impact, but continue through:
  - `resources/websites`
  - `resources/logos`
  - `resources/language`
  - root scripts (`default.py`, `addon.xml`, `changelog.txt`)
  - release/hash metadata
  - Kodi profile/cache behavior where relevant
- Identify repeated patterns that can be centralized without making the addon harder to debug:
  - request/header/session helpers
  - thumbnail caching
  - KVS parsing
  - pagination helpers
  - MP4/HLS probing
  - resolver fallback behavior
- Remove or archive only when the site/function is verified unused or broken beyond reasonable repair.
- Any cleanup must preserve Android/Fire TV compatibility.

## Priority A - Lib/Vendor Size Audit

- Audit `resources/lib/vendor` and identify what is actually used at runtime.
- Do not remove the current vendored stack blindly. First measure imports and site dependencies.
- Build a dependency map for:
  - `cloudscraper`
  - vendored `requests`
  - vendored `urllib3`
  - `requests_toolbelt`
  - `pyparsing`
  - `charset_normalizer`
  - any helper files used by protected sites
- Identify which websites currently require cloudscraper or the vendored session path.
- Create an experimental branch/work copy for a smaller `cloudscraper` profile:
  - `cloudscraper-mini`: remove clearly unused extras.
  - `cloudscraper-micro`: keep only modules needed by AdultHideout requests.
  - `cloudscraper-nano`: only attempt if mini/micro survive Windows and Android tests.
- Keep the original vendor folder restorable until the reduced stack passes KVAT and real Kodi tests.

## Guardrails

- No vendor reduction goes into release unless fallback behavior is clear.
- No broad cleanup goes into release without a before/after file list and targeted tests.
- No platform-specific optimization without Android verification.
- Keep zero-dependency policy: do not introduce new external Python package requirements.
- Keep the official hashes updated only after the final 1.0.14 release state is stable.
